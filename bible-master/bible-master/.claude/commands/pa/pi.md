# Post-Intake Command

**PM role.** After an engineer marks work as Done, the PM reviews, creates a task report on Outline, and announces to stakeholders. This command runs in the **personal assistant codebase** where the agent has access to communications (email, Teams, Slack) and Outline.

Always load `po` skill via `Skill("po")`.

**Cycle:** `/pp` (engineer pushes, marks Done) → `/pi` (PM announces to stakeholders)

---

## Input

```
/pi <plane-item-link-or-identifier>
```

The item should be a T1 (tier 1) work item in **Done** or **Passed** state. The PM has already reviewed the implementation and confirmed it passes QA.

---

## Process

### 1. Load Item Context

1. Run `plane-item-get.js` on the provided link/identifier to get the T1 work item
2. Verify the item is in **Done** or **Passed** state
3. If not in Done/Passed → stop and inform: "This item is in [state]. `/pi` runs on Done/Passed items after PM review."

### 2. Gather Full Context

Collect all available context for the task report:

1. **T1 description** — from plane-item-get.js output (includes Version doc link, Intake reference)
2. **T2 children** — run `plane-work-items.js` to get all tier 2 items with states, estimates, and descriptions
3. **Intake item** — if T1 has an `Intake: [PROJ-N](url)` reference, fetch via `plane-intake-get.js` for business context
4. **Version doc** — read from Outline (linked in T1 description) for implementation details
5. **Spec doc** — read module spec from Outline for current state context

### 3. Create/Update Task Report on Outline

Check the project's **Task Reports** folder (`OUTLINE_TASK_REPORTS_DOC_ID` from config) for an existing report for this item.

**Search:** Use `search_documents` with the item identifier (e.g., `PROJ-123`) to find existing task reports.

**If report exists** (item cycled back with additional T1s):
- Pull → Edit → Push to append new completion context

**If new report:**
- Create a new Outline document under the Task Reports folder

**Task Report Format:**

```markdown
# [T1 Title] — Task Report

> Item: [PROJ-N](plane_browse_url) | Version: [Outline](version_doc_url)
> Module: [module name] | Completed: [date]

## Summary

[2-3 sentence summary of what was delivered — written for non-technical stakeholders]

## Requirements (from Intake)

[Business context extracted from intake item — what was requested and why]

## Implementation

[Summary of what was built — extracted from version doc and T2 descriptions]

### Features Delivered

- [T2 title] ([PROJ-N]) — [one-line description]

### Points

| Feature | Estimate | Actual |
|---------|----------|--------|
| [T2 title] | N | N |
| **Total** | **N** | **N** |

## Screenshots / Evidence

<!-- PM: paste screenshots, demo recordings, or evidence below -->

---

*Report generated by `/pi`. Paste screenshots above before announcing.*
```

### 4. Prompt for Screenshots

After creating the scaffold, inform the user:

```
Task report created: [Outline link]

Please paste any screenshots or demo evidence into the report.
When ready, confirm to proceed with announcement.
```

**Wait for user confirmation** before proceeding to announcement.

### 5. Announce to Stakeholders

After user confirms the report is ready:

1. **Identify recipients** — read the intake item's "Requested by" field and any stakeholders from the PA's intel. **Always ask the user to confirm** the recipient list. Do not guess.
2. **Ask acknowledgement window** — "How long should recipients have to respond? (default: 24 hours)"
3. **Compose announcement email** — include:
   - Subject: `[Project] Feature Delivered: [T1 title]`
   - Body:
     - Brief summary of what was delivered
     - Link to Outline task report (full details + screenshots)
     - Link to Plane intake item (tracking)
     - Acknowledgement note: "If you have concerns, please respond within [N hours]. Otherwise, this will be deployed to production."
4. **Send via workspace comms MCP** — use the workspace's `comms_mcp` from config
5. **Move item to Announced** — `plane-item-update.js PROJ-N --state announced`

### 6. Output

```
Post-Intake Complete
Item: [PROJ-N] [title]
Task Report: [Outline link]
Announced to: [recipient list]
Acknowledgement window: [N hours]
Status: Announced

Next: After acknowledgement window, move to Approved → chase engineer to deploy → mark Deployed.
```

## Post-Completion States

After all T1s under an intake are Done, the intake auto-closes to Done. The PM then manages the post-completion pipeline on the **intake item**:

```
Done → Passed → Announced → Approved → Deployed
       (manual)  (/pi)       (manual)   (manual)
```

| State | Who | How | Purpose |
|-------|-----|-----|---------|
| **Done** | Auto | `/pp` auto | All T1s complete, intake auto-closed |
| **Passed** | PM | Manual drag in Plane | PM reviewed, confirmed QA pass |
| **Announced** | PM | `/pi` command | Stakeholders notified, in feedback window |
| **Approved** | PM | Manual drag in Plane | Stakeholder acknowledged, ready to deploy |
| **Deployed** | PM | Manual drag in Plane | Deployed to production, lifecycle complete |

**T1 terminal state is Done.** All states beyond Done belong to the intake item only.

**Passed** is optional — `/pi` can shortcut from Done directly to Announced.

**If work doesn't pass** (Done/Passed/Announced): PM uses `/pm improve` in the project codebase to extend the intake with new T1(s). See `/pm` IMPROVE protocol.

---

## Critical Rules

1. **Item must be Done or Passed** — `/pi` does not run on in-progress items
2. **Always confirm recipients** — never guess who gets the announcement
3. **Wait for screenshots** — scaffold first, announce after user confirms report is complete
4. **Check for existing report** — items may cycle back; append, don't duplicate
5. **Use workspace comms MCP** — route email through the correct workspace's communication tools
6. **Default 24hr window** — always state the acknowledgement deadline in the email
7. **Include both links** — Outline task report (details) + Plane intake (tracking) in every announcement
