# Git Push with Smart Grouping

Always load `po` skill via `Skill("po")`.

## Step 0: Gather Task Context

Build a context map from **Plane** (work items + modules), **Outline** (spec docs), and **plan files** (in-progress implementation plans).

### 0.1 Load MCP Tools

Use the Plane and Outline MCP server names from the `po` skill. Load in parallel:

```
ToolSearch("select:mcp__{plane_server}__list_cycles")
ToolSearch("select:mcp__{plane_server}__list_cycle_work_items")
ToolSearch("select:mcp__{plane_server}__list_modules")
ToolSearch("select:mcp__outline__search_documents")
```

### 0.2 Find Current Cycle

Calculate current ISO week → format as `{YYYY}/{WW}`.

Use `list_cycles` with the WORKSPACE_SLUG and PROJECT_ID from the `po` skill.

Find the cycle named `{YYYY}/{WW}` → extract `cycle_id`.

If no matching cycle exists, skip to Step 0.4 (proceed with plan files + Outline only).

### 0.3 List Cycle Work Items + Modules

Run in parallel:

**Work items:**

Use `list_cycle_work_items` with WORKSPACE_SLUG, PROJECT_ID, and cycle_id from the `po` skill.

**Large response handling:** This call typically auto-saves to a `.txt` file. Parse with:

```bash
python -c "
import json, sys
with open('<SAVED_FILE_PATH>', 'r', encoding='utf-8') as f:
    data = json.load(f)
items = data['result']
parents, children = {}, {}
for it in items:
    pid = it.get('parent')
    if pid:
        children.setdefault(pid, []).append(it['name'])
    else:
        parents[it['id']] = it['name']
for uid, name in sorted(parents.items(), key=lambda x: x[1]):
    print(name)
    for child in children.get(uid, []):
        print(f'  - {child}')
    print()
"
```

Replace `<SAVED_FILE_PATH>` with the actual path from the MCP response. Use forward slashes.

**Modules:**

Use `list_modules` with WORKSPACE_SLUG and PROJECT_ID from the `po` skill.

Extract module names. These provide high-level feature domains.

### 0.4 Scan Context Files

Check for active tier 1 context:

```
Glob("cycles/**/context.md")
```

For each context file found, read it to extract:

- **Tier 1 scope** (module name + version from `## Tier 1:` heading)
- **Tier 2 items table** (feature names, states, point estimates)
- **Completion status** (how many tier 2s are Done/In Progress)

Context files are generated by `plane-work-items.js` and contain condensed tier 1 + tier 2 summaries.

### 0.5 Read Outline Specs (If Modules Match Changes)

After identifying which modules are relevant to the current changes (from Step 1), read their Outline spec docs for richer context:

Use `search_documents` with the module name and COLLECTION_ID from the `po` skill.

Use the Non-Technical Description from each spec doc to write better commit messages that describe **behavior**, not just file paths.

### 0.6 Build Context Map

Combine all sources into a context map:

- **Context files** (highest priority) — tier 1 scope and tier 2 feature inventory with states
- **Work items** — active tasks in the current cycle: `[vX.Y.Z | Module] Title > Feature`
- **Modules** — feature domains
- **Outline specs** — behavioral descriptions of each module

Extract the module scope from each (e.g., `"3D Scene"` from `"[v1.0.0 | 3D Scene: Canvas] Camera system > Camera controls"`).

## Step 1: Analyze Current Changes

Run `git status` and `git diff` to understand all current changes.

---

## Step 2: Match Files to Tasks

For each changed file:

### 2.1 Try to Match to Context

Use the context map to match files. Priority order:

1. **Context file match** — if a context file's tier 2 feature names align with the changed file's purpose, group with that feature
2. **Work item match** — use title keywords and scope alignment
3. **Module match** — match by file path patterns and module names

### 2.2 Fallback for Unmatched Files

If no match found:

1. Read the file content and diff
2. Investigate how the file is used in the codebase
3. Understand the purpose of the change
4. Group with similar unmatched changes

---

## Step 3: Create Commits

For EACH group of changes:

1. Stage the specific files for this group using `git add`
2. Create a descriptive commit message:
    - Use behavioral language when Outline context is available (e.g., "camera recording playback controls" not "update useTimeController")
    - Reference the feature/module scope naturally (e.g., "shot list preview caching" not "modify PageShotLists")
3. Commit the changes
4. Move to the next group

---

## Critical Rules

- Ignore `tmpclaude-*` files/folders — never stage or commit them
- Include `cycles/` files (context.md, plan files) — always stage and commit alongside related code changes
- Create all commits BEFORE pushing
- Push only once at the end (not after each commit)
- Never mention Claude in commits (no co-authors, no "Generated with Claude")
- Stay on current branch unless explicitly instructed otherwise
- Use clear, professional commit messages describing the changes
- Keep commit messages simple — no task references, project identifiers, or URLs
- Run git commands directly (e.g., `git status`, `git add`, `git commit`)
